## Core pipeline (how a PDF becomes classified sheet rows)

1. **`extract_statement.py`** — parses a bank PDF into transaction rows using **word-position-based column detection** (not pdfplumber's table-detection, which is unreliable). Key functions:
   - `_cluster_lines()` groups words into visual rows by y-position.
   - `_detect_header_columns()` finds the header row via a description-keyword anchor, maps header tokens to fields, computes column x-ranges as **midpoints between adjacent column anchors**.
   - `_bucket_line()` assigns each word to a field bucket by x-position (x0 for text fields, x1 for right-aligned amount fields).
   - A **"bank profile" system** (`config/bank_profiles.json`) lets a new bank's different header wording, footer noise, or misaligned column positions be handled via config instead of new code: `extra_header_tokens`, `extra_exclude_patterns`, `column_anchor_overrides` (manual x-position for a field), `excluded_fields` (drop a field from detection entirely — for a column with a header label but zero real data, which would otherwise steal width from a neighboring column).
2. **`run_pipeline.py`** — orchestrates: unlock PDF → `step_extract()` → upload to sheet → classify.
3. **`classify_transactions.py`** — the classification rule engine (`resolve_business_fields()` / `_resolve_business_fields()`). Assigns HEAD, BUSINESS UNIT, TYPE FOR RERA IDW, TCP Head to each row via a strict waterfall of ~12 numbered rules (Rule 0 Manual Override → Rule 1 internal-transfer-by-account-number → Rule 2 BOM/MAHB IFSC → Rule 2b linked-account IFSC → Rule 3 cheque deposit → Rule 4 incoming-payment Collection → Rule 6 Beneficiary Master lookup → Rule 7 salary keyword → Rule 8 EPF/ESI → Rule 8b TDS/Professional Tax → Rule 9 Marketing → Rule 10 Bank Charges → Rule 10b Tax → Rule 10c Electricity → Rule 11 role keyword → ... falls back to RAG AI). First matching rule wins, returns immediately.
4. **`upload_to_sheets.py`** — writes/dedups rows into the account's worksheet tab. Dedup key is **Date + Debit + Credit** (deliberately excludes Description) — meaning if extraction logic changes, re-uploading the same PDF will NOT overwrite already-uploaded rows with corrected text; it'll just skip them as "duplicates". To fix already-uploaded bad data, you must delete the tab and re-upload, or directly patch cells.
5. **`web_app.py`** — Flask app: Dashboard, Manual Upload, Check Bank Emails, History, Admin (Account Passwords, Company Sheet Links, Account Sheet Links), Beneficiary Master, Gmail Accounts.

## Known recurring quirk (not a bug in your code — a Vercel platform limitation)

Vercel serverless functions sometimes don't reliably continue a `threading.Thread`-based background job after the initiating HTTP request returns. Symptom: the Dashboard progress bar freezes at some %, but the underlying pipeline often still completes successfully in the background. **Always verify directly against the Google Sheet or History** (via a `gspread` query) rather than trusting a stuck progress bar — this has happened repeatedly and the data usually did land correctly.

## Established working patterns / feedback to keep following

- **New bank onboarding**: use the `config/bank_profiles.json` escape hatches first; only touch `extract_statement.py` code if the profile system genuinely can't express the difference. Always do a full-row diff (all 7 columns) against a real sample PDF before trusting it — a "looks right" spot-check missed a real bug once (see item 3 above).
- **Classification rules**: numbered rules in `classify_transactions.py`, first-match-wins waterfall. When adding a new rule, follow the existing pattern of requiring a specific, narrow signal (not just a loose keyword) to avoid false positives — see the docstring on `_find_bom_internal_ifsc` for the documented history of two real bugs caused by matching too loosely.
- **Don't guess when the ground truth itself is ambiguous or inconsistent** — leave the field flagged `"?"` with a REASON note rather than picking a default that's sometimes wrong. The user has explicitly confirmed this preference twice this session (Business Unit for BOM 905, and rejecting the cross-account matching heuristic once shown it produces false positives).
- **Verify against the accounts team's real reference data**, not just "the code runs without error." The user has a reference Excel workbook (`Copy of DPL Bank Statements 2026-27 (4).xlsx`) with per-account tabs that serve as ground truth for classification — ask for it again if similar work comes up for another account.
- **Deploy cycle**: `git add` → `git commit` (message ending `Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>` if using Claude) → `git push origin main` → `vercel --prod --yes` (can take 2-3 min; poll via `vercel inspect <url>` if the CLI call itself errors with a network blip — the deployment often succeeds anyway) → `curl -s -o /dev/null -w "status: %{http_code}\n" https://bank-statement-automation-bai4.vercel.app` (expect `302`, a login-redirect, confirming the site is up).
- **New Supabase tables**: this project never runs schema migrations in code — every table is created directly in the Supabase SQL editor by the user, then a plain Python store module (mirroring `company_sheets_store.py` / `account_sheet_links_store.py`) reads/writes it. Ask the user to run the `CREATE TABLE` statement before deploying code that depends on it.
